# CLAUDE.md — инструкции для работы над книгой «1С как иностранный язык»

## Контекст проекта

Учебный курс для абсолютных новичков без опыта программирования. Главная идея — рассматривать 1С как **язык описания и преобразования данных в деловой системе**, а не как набор синтаксических конструкций. Несущий приём — метафора иностранного языка.

- **Метаданные:** `metadata.yaml`
- **Статус и план:** `docs/plan.md`
- **Полный план курса:** `docs/00_plan_kursa.md`
- **Обложка:** `assets/img/cover.png`
- **Главы:** `chapters/`

## Структура файлов

```
chapters/NN_nazvanie/                     — папка модуля
chapters/NN_nazvanie/NN-MM_paragraf.md    — файл параграфа
```

Примеры:
```
chapters/00_vvedenie/00-01_chto_takoe_dannye.md
chapters/01_leksika/01-03_komanda_soobschit.md
```

Заголовок модуля (`# Название`) в `.md` файлах **не пишем** — он генерируется из имени папки при сборке. Параграфы начинаются с `##`.

## Аудитория

Абсолютный новичок без опыта программирования. Это диктует жёсткие правила:
- Нельзя опираться на интуицию из других языков
- Объяснять не только *что*, но и *почему так вообще можно*
- Каждое новое понятие — концептуальный сдвиг, а не просто термин
- Курс медленный на старте и ускоряется к концу

## Формат параграфа

Каждый параграф устроен одинаково:

1. **Открытие** — связь с предыдущим параграфом, постановка нового вопроса.
2. **Содержательные разделы** (5–9 штук).
3. **Главное в одном абзаце** — плотный пересказ ключевых мыслей в устной форме (не tl;dr, а смысловой повтор).
4. **Контрольные вопросы** (5–7 штук).
5. **Упражнения** (4–6 штук).
6. **Ответы, указания, разборы** — три уровня самопроверки.
7. **Концовка** — мостик к следующему параграфу.

Внутри содержательных разделов:
- 1–3 ключевых мысли оформляются выносками-якорями (`>`)
- Тяжёлые понятия — бытовой пример + пример на 1С
- Где уместно — ASCII-схема
- Где есть код — таблица состояния («прогонка»)

## Стилистические правила

- **Никаких абсолютов** «без X нет Y». Формулировать через «возникает, когда…», «требует…». Это касается и технических утверждений: «не может стать бесконечным» — тоже абсолют; точнее: «риск значительно ниже» или «не требует ручного управления завершением».
- **Никакого антропоцентризма.** Если правило работает и для человека, и для программы — формулировать сразу для обоих.
- **Метафоры — с парной строгой формулировкой.** «Данные молчат — а если строже…»
- **Живые персонификации сохраняем** («данные ждут», «компьютер не догадывается»). Но рядом — строгая версия.
- **Бытовой регистр + строгий термин.** «Отобрать (отбор, фильтрация)». Сначала бытовое слово, потом термин в скобках.
- **Длинные связные фразы допустимы** — это интонация классического учебника. Рубленый стиль — нет.

## Структурные правила

- **Ключевые мысли как якоря.** Если фраза стоит цитирования — оформлять выноской `>` сразу.
- **Термины до их официального ввода** — с оговоркой: «слово X здесь стоит мимоходом, без объяснения; вернёмся к нему в § Y».
- **Связки между параграфами явные.** Каждый параграф открывается напоминанием предыдущего и закрывается мостиком к следующему.
- **Возврат к ранее введённым правилам-якорям** — упоминать явно, чтобы выстраивалась преемственность.
- **Нет «висячего знания».** Не вводить понятие, синтаксис или термин, который в этом параграфе нигде не используется. Если нужно упомянуть — только мимоходом, с явной отсылкой к тому параграфу, где оно будет раскрыто.
- **Одна новая ось сложности на параграф.** Не вводить несколько незнакомых механизмов одновременно. Если параграф вводит новую конструкцию — примеры строятся на уже знакомом материале.
- **Когнитивная нагрузка параграфов чередуется.** После плотного параграфа следующий должен быть заметно легче — меньше разделов, проще примеры. Это не слабость, а методический выбор.

## Эстетика

- **ASCII-схемы как основа.** Простые, моноширинные, в рамках кодовых блоков.
- **Mermaid** — только когда ASCII не справляется.
- **Минимум разметки, максимум прозы.** Списки, таблицы, выноски — там, где они работают, а не везде.

## Чего не делать

- Не превращать учебник в IT-блог: без многочисленных подзаголовков, плакатных блоков.
- Не использовать эмодзи в основном тексте.
- Не дублировать системы выделения. Якорь — выноска `>`. Термин — **жирный** или *курсив* по контексту.
- Не трогать `book.md` — временный файл, генерируется при сборке.
- Не коммитить папку `output/`.

## Стандарты оформления кода (с § 2.3 — обязательно)

- После каждой команды (присваивание, вызов процедуры) — `;`
- После `КонецЕсли`, `КонецЦикла`, `КонецПроцедуры` — `;`
- После `Тогда`, `Иначе`, `ИначеЕсли` — *без* `;`
- После строки только с комментарием — *без* `;`
- Комментарии — через `//`

В параграфах Модулей 0–2 и § 2.1–2.2 — массовая правка в финальной редактуре.

## Рабочий процесс

1. Параграф пишется **целиком**, потом обсуждается. Не дробить на куски — теряется связность.
2. После параграфа — внешний разбор. Замечания по содержанию рассматриваются всегда; замечания по подаче — с осторожностью.
3. Правки вносятся **точечно**, не переписыванием параграфа.
4. После значительных правок — финальная сверка на расхождения с другими параграфами.
5. При вводе новой методики — обратная сверка с уже написанным материалом.

## Что требует проверки в финальной редактуре

- `Выбор Из ... Когда ... КонецВыбора` — в языке программирования 1С **отсутствует** (только в языке запросов). Из § 3.1 убрана.
- `СтатусСообщения.Информация` (§ 1.3) — проверить точный синтаксис.
- Формат даты-литерала `'ГГГГММДД'` (§ 2.2) — проверить.
- Поведение `Истина И Ложь`, `"5" + 3`, склеивание со строкой логического значения (§ 2.2).
- `ТипЗнч()`, `Тип("Число")` (§ 2.2) — синтаксис обращения.

**Правило:** каждый листинг и каждое утверждение о поведении платформы проверять по синтакс-помощнику или официальной документации, а не принимать на веру.

## Устоявшиеся терминологические решения

Эти определения зафиксированы в тексте и должны соблюдаться во всех последующих параграфах:

- **Коллекция** — это *значение* особого рода, способное хранить внутри себя несколько других значений. **Не** «переменная, которая хранит несколько значений» — это смешивает имя и значение, что противоречит § 1.1.
- **Переменная цикла** в «Для каждого Из» — одна и та же переменная на всех итерациях; в неё помещается очередной элемент коллекции, как при обычном присваивании.
- **Итерация** — один полный проход через тело цикла (введено в § 3.2).

## Воркфлоу рецензирования (DeepSeek)

Параграфы рецензируются автоматически через DeepSeek API.

### Файлы

- `scripts/review.sh <файл>` — отправить параграф на рецензию
- `scripts/status.sh` — таблица статусов всех параграфов
- `scripts/status.sh detail <файл>` — история рецензий конкретного параграфа
- `scripts/status.sh set <файл> approved|needs_revision|skip [примечание]` — выставить решение вручную
- `scripts/prompts/reviewer.md` — системный промпт рецензента (редактировать при необходимости)
- `.env` — API-ключ (не в git; шаблон: `.env.example`)

### Хранение результатов

Рецензии сохраняются в `reviews/<модуль>/<параграф>.<модель>.<дата>.md` и **не попадают в git** (папка `reviews/` в `.gitignore`). Трекер статусов `docs/review-status.json` тоже gitignored — это локальный рабочий файл.

### Безопасность

API-ключ читается Python-скриптом напрямую из `.env` — через shell-окружение он не передаётся. Файл `.env` добавлен в `.gitignore`. В репозиторий попадают только: `scripts/review.sh`, `scripts/status.sh`, `scripts/prompts/reviewer.md`, `.env.example`.
